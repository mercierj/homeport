## Downgrade Or Relabel

Use explicit compatibility levels alongside the old operational status:

- L0 for generated migration/runbook/app-change artifacts only.
- L1 for deployable backend provisioning without provider API compatibility.
- L2 for partial provider-shaped adapters or generated client patches.
- L3 only when official provider SDK/REST contract tests prove the supported surface, including error shapes and lifecycle basics.
- L4 only when L3 plus `Authorize(principal, action, resource, context)`, allow/deny tests, quotas, audit events, idempotency, pagination, backup, validate, cutover, and rollback are all runnable and passing.

The ledger schema now supports this directly through `compatibility_level`; treat every current `status: full` row with `compatibility_level` below L4 as "operational checklist full, provider-grade compatibility unproven."
